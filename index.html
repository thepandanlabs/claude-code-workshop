Why This Works

The theory behind the three-file discipline. What Claude Code actually does — and what a chat box structurally cannot.

The mental model — the loop

Claude Code is an agentic loop. Each turn: Claude reads your context files (CLAUDE.md, PRD.md, files opened so far) → acts through tools → waits for your review. Context is durable on disk and re-read every turn. You steer Claude by editing files, not by arguing in chat. In Plan Mode, the edit and bash tools are disabled at the framework level — Claude physically cannot write anything until you approve a plan you actually read.

Thesis 1

Structured intent beats clever prompting

The model fills every gap with an assumption. PRD.md + CLAUDE.md fill those gaps before the first line of code. Once they do, the good prompt is short — the structure carries the load.

Thesis 2

Verification makes the demo real

An eval is a saved input, the known-right answer, and an automatic check that they match. "Looks right in the demo" is not a test. The eval suite is the compounding asset — it catches regressions automatically, on every change, forever.

Thesis 3

Collaborator, not a genie

Claude Code arrived this morning with zero context about your project. A clear brief gets exceptional work. A vague wish gets reasonable guesses that compound into a codebase you didn't want. Brief it. Read the plan. Correct via files, not arguments.

What a chat box structurally cannot do
files

Iterate a folder, including files that arrive tomorrow. No chat primitive for that.

durable state

ledger.db persists and dedupes across every run. Chat sessions forget.

determinism

Same input → byte-identical output, every time. That's what makes regression tests possible.

composability

Pipe to cron, Makefile, MCP, or email. A chat tab has no equivalent.
